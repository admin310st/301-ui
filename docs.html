301.st Documentation

Open product guides for the parts of 301.st that are already shipped and stable.

301.st is a Cloudflare-first control plane for teams that manage domains, projects, sites, and redirects in one place.

Read getting started guide About 301.st

Core Concepts

Integrations connect the platform to your real infrastructure

Integrations are the access layer. They let 301.st read the systems you already operate instead of asking you to rebuild them inside a separate hosting panel.

Today that mainly means Cloudflare and registrar-linked workflows, using credentials you control and can revoke at any time.

Domains turn raw inventory into an operational workspace

The domains layer is where synced inventory becomes usable. Instead of a loose list of names, you get status review, filtering, inspection, and day-to-day actions from one table.

That makes domains the working foundation for everything downstream: projects, sites, redirects, and TDS bindings.

Projects and sites give structure to the rest of the stack

Projects group related work, while sites attach destinations and operational context. Together they stop the account from becoming one flat surface with no boundaries.

That structure matters once you manage multiple offers, traffic paths, or teams and need routing logic to stay readable.

Redirects are the most mature shipped traffic workflow

Redirects are where the current product is the most operationally settled. You can create, review, and sync redirect rules without hopping through provider dashboards for every change.

For most new users, this is the clearest production workflow after integrations, domain sync, and basic project structure.

TDS is shipped, but still newer than the core redirect layer

TDS exists in the product as a site-scoped routing surface with rules, presets, bindings, and Cloudflare-backed execution.

Public docs keep it framed more carefully because it is still a younger surface than redirects and is better understood as an active product area than as the main entry workflow.

What These Pages Cover

This public documentation is written for operators, founders, and teams evaluating 301.st. It focuses on product workflows, not internal implementation details.

  • Included: setup flow, integrations, domains, projects, sites, redirects, product status pages
  • Not included: internal engineering docs, source-level implementation notes, and unreleased feature promises
  • Audience: operations teams, affiliate stacks, domain managers, and Cloudflare-centric workflows

Internal platform specs still live in repository docs and the API wiki. The public pages are meant to be readable, indexable, and accurate to what is already live.

FAQ

Is this docs section public?

Yes. These pages are meant to be readable without login and safe to index.

Does it describe only shipped features?

That is the goal. Stable workflows are documented as guides; TDS is kept on a separate status page until rollout is complete.

Where should technical contributors look?

For engineering-level details, use the repository README, style guide, and API wiki rather than the public docs pages.

Will more pages be added?

Yes. The section is structured so new stable product areas can be added without turning /docs into a single oversized page.
