Security

How we protect your infrastructure credentials and configuration data.

Found a Security Issue?

We take security seriously. If you discover a vulnerability, please report it responsibly:

  • Email: security@301.st (PGP key available on request)
  • What to include: Steps to reproduce, affected endpoints/pages, potential impact
  • Response time: We'll acknowledge within 48 hours and provide updates every 5 business days
  • Disclosure timeline: We request 90 days to fix critical issues before public disclosure

Bug bounty: Currently informal. We'll publicly credit researchers (with permission) and may offer rewards for critical findings once funding allows.

1. Credential Security

User Passwords

  • Hashing: Bcrypt or Argon2 with per-user salts. No plaintext storage.
  • Minimum requirements: 8 characters, mix of uppercase/lowercase/digits enforced client-side and server-side.
  • Password reset: Time-limited tokens (30 min expiry) sent via email. CSRF protection on reset forms.

API Tokens (Cloudflare, Registrars)

  • Encryption at rest: AES-256-GCM with unique keys per token. Encryption keys stored in secure vault (e.g., Hashicorp Vault, AWS KMS).
  • Scoped permissions: We request minimum required scopes (e.g., "Zone:Read, DNS:Edit" for specific zones, not global "Account:Edit").
  • Token rotation: You can regenerate tokens at any time. Old tokens are immediately revoked in our system.
  • Revocation monitoring: We detect revoked tokens and alert you if connectivity fails.

Session Management

  • Access tokens: Short-lived JWT (15 min expiry), stored client-side in memory (not localStorage).
  • Refresh tokens: HTTP-only, secure, SameSite cookies. Stored encrypted server-side. 30-day expiry (sliding window).
  • Token rotation: Every refresh request issues a new access token and invalidates the old refresh token.
  • Logout: Immediately revokes refresh token server-side and clears client session.

2. Infrastructure Security

  • TLS 1.3: All traffic encrypted. HSTS enabled (HTTP Strict Transport Security).
  • CSP (Content Security Policy): Strict CSP headers to prevent XSS attacks.
  • Cloudflare Workers: Frontend served from edge network. DDoS protection and WAF enabled.
  • Database: PostgreSQL with row-level security (RLS). Encrypted at rest. Read replicas for analytics queries.
  • Secrets management: Environment variables injected at runtime. No secrets in code or Git history.
  • Dependency scanning: Automated checks for vulnerable npm packages (Dependabot, Snyk).

3. Access Control

  • Principle of least privilege: Application services can't access encryption key vault directly (keys fetched via separate auth layer).
  • Database credentials: Read-only replicas for analytics. Write access restricted to API server with audit logging.
  • Admin access: 2FA required for production database/server access. SSH keys rotated quarterly.
  • Audit logs: All privileged operations logged (token creation, domain sync, permission changes). Logs stored 90 days.

4. Application Security

  • Input validation: All API inputs validated server-side. SQL injection prevented via parameterized queries (no raw SQL).
  • CSRF protection: State tokens for password reset, account deletion, and other sensitive operations.
  • Rate limiting: Login attempts (5/min), API requests (100/min per user), token refresh (10/hr).
  • Turnstile: Cloudflare's CAPTCHA alternative on login/registration to prevent bot abuse.
  • XSS prevention: All user-generated content sanitized. CSP headers block inline scripts.
  • Subdomain isolation: User content (if any) served from separate subdomain to prevent cookie theft.

5. Third-Party API Security

When we call Cloudflare, registrar, or other third-party APIs on your behalf:

  • Token scoping: We request minimum required permissions and never request global "Account:Edit" unless absolutely necessary.
  • Request logging: We log API endpoint, timestamp, HTTP status. We never log request bodies containing tokens or sensitive data.
  • Error handling: API errors sanitized before showing to users (no stack traces with internal paths).
  • Retry logic: Exponential backoff for failed requests. Respect rate limits (429 responses).

6. Incident Response

If a security breach occurs:

  • Detection: Automated alerts for unusual activity (mass token revocations, failed login spikes, database anomalies).
  • Containment: Immediately revoke compromised tokens, lock affected accounts, isolate vulnerable services.
  • Notification: Email affected users within 72 hours (GDPR requirement). Provide details on what data was accessed.
  • Post-mortem: Public incident report (anonymized) within 30 days. Outline root cause, mitigation steps, prevention measures.

Status page: We'll maintain a public status page (status.301.st) for uptime and security incident tracking.

7. Compliance & Certifications

Current compliance:

  • GDPR: Data minimization, right to erasure, data portability (see Privacy Policy).
  • CCPA: California privacy rights (data access, deletion) supported.

Future certifications (roadmap):

  • SOC 2 Type II (when we have 10+ enterprise customers)
  • ISO 27001 (information security management)
  • Penetration testing by third-party firm (annual)

8. User Security Best Practices

We can only secure our side. Here's how you can protect your account:

  • Strong passwords: Use a password manager. Don't reuse passwords across sites.
  • 2FA (coming soon): Enable two-factor authentication when we launch it.
  • API token hygiene: Revoke unused tokens in Cloudflare/registrar dashboards. Rotate tokens quarterly.
  • Monitor activity: Check audit logs in account settings for unexpected API calls.
  • Phishing awareness: We'll never ask for your password via email. Always check URLs (app.301.st).
